## Раздел 1. Анализ предметной области

Система онлайн-записи предназначена для автоматизации работы агентства недвижимости по организации показов объектов (квартир, домов, апартаментов) потенциальным покупателям. Клиент выбирает интересующий его объект из базы агентства и записывается на удобное время для его осмотра. За каждым объектом закреплён ответственный риелтор, который сопровождает сделку и присутствует на показе, поэтому при создании записи автоматически определяется специалист. Помимо основной функции подбора времени, система позволяет клиентам сохранять понравившиеся варианты в список избранных объявлений для последующего сравнения и быстрого доступа. База данных должна поддерживать следующие бизнес-правила: один и тот же объект недвижимости не может быть назначен к показу двум разным клиентам в одно и то же время, что исключает накладки в расписании риелторов; при попытке удаления клиента, у которого есть активные или архивные просмотры, операция должна быть заблокирована для сохранения истории взаимодействий; цена и площадь объекта не могут быть отрицательными или нулевыми, а количество комнат должно быть целым положительным числом; все контактные данные клиентов и риелторов должны быть уникальными во избежание дублирования записей; каждая запись на просмотр должна быть обязательно связана с существующим клиентом, объектом и риелтором, что обеспечивает ссылочная целостность внешних ключей; статус просмотра может принимать строго определённые значения, отражающие жизненный цикл сделки: от планирования до успешного закрытия или отмены.

## Раздел 2. Концептуальная модель

Концептуальная модель базы данных агентства недвижимости включает пять сущностей: Клиент, Риелтор, Объект недвижимости, Просмотр и Избранное. Сущность Клиент описывает физическое лицо, желающее приобрести недвижимость, и содержит атрибуты: идентификатор клиента (первичный ключ), фамилия, имя, телефон и адрес электронной почты. Сущность Риелтор представляет сотрудника агентства, ответственного за ведение объектов и проведение показов, и включает атрибуты: идентификатор риелтора (первичный ключ), фамилия, имя, телефон и дата найма. Сущность Объект недвижимости описывает жилое помещение, выставленное на продажу, и содержит атрибуты: идентификатор объекта (первичный ключ), адрес, тип объекта (квартира, дом или апартаменты), количество комнат, общая площадь в квадратных метрах и цена в рублях. Сущность Просмотр является центральной в системе и фиксирует факт записи клиента на осмотр конкретного объекта в определённое время; её атрибуты включают идентификатор просмотра (первичный ключ), дату и время проведения, статус (запланирован, проведён, сделка или отменён), а также дату создания записи. Сущность Избранное представляет связь между клиентом и понравившимися ему объектами и содержит составной первичный ключ из идентификатора клиента и идентификатора объекта, а также дату добавления в избранное. Между сущностями установлены следующие связи. Связь между Клиентом и Просмотром имеет тип «один-ко-многим», поскольку один клиент может записаться на множество просмотров, но каждый конкретный просмотр принадлежит ровно одному клиенту. Связь между Риелтором и Просмотром также «один-ко-многим»: один риелтор проводит множество показов, но каждый показ ведёт один конкретный специалист. Связь между Объектом недвижимости и Просмотром — «один-ко-многим»: один объект может быть показан многократно разным клиентам в разное время, но каждая запись относится к одному конкретному объекту. Связь между Риелтором и Объектом недвижимости — «один-ко-многим»: за каждым объектом закреплён один ответственный риелтор, но один риелтор может вести несколько объектов одновременно. Связь между Клиентом и Объектом недвижимости реализована как «многие-ко-многим» через промежуточную таблицу Избранное: один клиент может добавить в избранное несколько объектов, и один объект может находиться в избранном у нескольких клиентов. ## Раздел 3. Логическая модель и нормализация

Реляционная схема базы данных включает пять таблиц, соответствующих сущностям концептуальной модели. Таблица clients содержит столбцы: client_id типа INT с автоинкрементом, ограничениями PRIMARY KEY и NOT NULL; last_name типа VARCHAR(50) с ограничением NOT NULL; first_name типа VARCHAR(50) с ограничением NOT NULL; phone типа VARCHAR(20) с ограничениями NOT NULL и UNIQUE, а также проверочным ограничением CHECK, требующим соответствия формату российского номера; email типа VARCHAR(100) с ограничениями NOT NULL и UNIQUE. Таблица realtors включает столбцы: realtor_id типа INT с автоинкрементом, PRIMARY KEY и NOT NULL; last_name типа VARCHAR(50) с NOT NULL; first_name типа VARCHAR(50) с NOT NULL; phone типа VARCHAR(20) с NOT NULL и UNIQUE; hire_date типа DATE с NOT NULL и значением по умолчанию CURRENT_DATE. Таблица properties состоит из столбцов: property_id типа INT с автоинкрементом, PRIMARY KEY и NOT NULL; address типа VARCHAR(255) с NOT NULL; property_type типа ENUM со значениями «квартира», «дом», «апартаменты» и ограничением NOT NULL; rooms типа TINYINT UNSIGNED с NOT NULL и CHECK, требующим положительного значения; area типа DECIMAL(7,2) с NOT NULL и CHECK, требующим положительного значения; price типа DECIMAL(12,2) с NOT NULL и CHECK, требующим положительного значения; realtor_id типа INT с NOT NULL, являющийся внешним ключом, ссылающимся на таблицу realtors с поведением ON DELETE RESTRICT и ON UPDATE CASCADE. Таблица viewings является центральной таблицей записей на просмотр и включает столбцы: viewing_id типа INT с автоинкрементом, PRIMARY KEY и NOT NULL; client_id типа INT с NOT NULL, внешний ключ к таблице clients с ON DELETE RESTRICT и ON UPDATE CASCADE; property_id типа INT с NOT NULL, внешний ключ к таблице properties с ON DELETE RESTRICT и ON UPDATE CASCADE; realtor_id типа INT с NOT NULL, внешний ключ к таблице realtors с ON DELETE RESTRICT и ON UPDATE CASCADE; viewing_datetime типа DATETIME с NOT NULL; status типа ENUM со значениями «запланирован», «проведён», «сделка», «отменён» и NOT NULL со значением по умолчанию «запланирован»; created_at типа TIMESTAMP со значением по умолчанию CURRENT_TIMESTAMP; кроме того, на таблицу наложен составной уникальный ключ по паре столбцов property_id и viewing_datetime, а также созданы индексы для ускорения поиска по client_id и viewing_datetime. Таблица favorites реализует связь «многие-ко-многим» между клиентами и объектами и содержит столбцы: client_id типа INT с NOT NULL, являющийся частью составного первичного ключа и внешним ключом к clients с ON DELETE CASCADE и ON UPDATE CASCADE; property_id типа INT с NOT NULL, вторая часть составного первичного ключа и внешний ключ к properties с ON DELETE CASCADE и ON UPDATE CASCADE; added_at типа TIMESTAMP со значением по умолчанию CURRENT_TIMESTAMP. Процесс приведения к третьей нормальной форме был проведён последовательно для каждой таблицы. Первая нормальная форма требует атомарности значений в столбцах, что соблюдается во всей схеме: каждый столбец содержит ровно одно значение, отсутствуют повторяющиеся группы и массивы, а многозначные связи вынесены в отдельную таблицу favorites. Вторая нормальная форма требует, чтобы каждый неключевой столбец зависел от полного первичного ключа, что актуально для таблицы favorites с составным ключом: столбец added_at зависит от комбинации client_id и property_id, то есть от факта добавления конкретным клиентом конкретного объекта, а не от части ключа; для остальных таблиц с простым первичным ключом условие выполняется автоматически, так как все неключевые столбцы зависят от единственного идентификатора. Третья нормальная форма исключает транзитивные зависимости, когда неключевой столбец зависит от другого неключевого столбца: в таблице properties столбцы rooms и area зависят непосредственно от первичного ключа property_id, а не от адреса или цены; в таблице viewings статус зависит от самого просмотра, а не от даты или клиента; стоимость объекта не дублируется в таблице просмотров, а хранится единожды в properties и при необходимости получается через операцию JOIN; дата найма риелтора в таблице realtors зависит от самого риелтора, а не от его телефона или фамилии. Денормализация в схеме отсутствует, так как все таблицы полностью соответствуют третьей нормальной форме, что исключает аномалии обновления, вставки и удаления данных и обеспечивает целостность хранимой информации.

## Раздел 4. SQL-скрипт создания базы данных

```sql
-- ============================================================
-- ВАРИАНТ 17. Система записи в агентство недвижимости на просмотр квартир
-- База данных: real_estate_viewing_variant17
-- СУБД: MySQL 8.0
-- Кодировка: utf8mb4, collation: utf8mb4_unicode_ci
-- ============================================================

DROP DATABASE IF EXISTS real_estate_viewing_variant17;
CREATE DATABASE real_estate_viewing_variant17
    CHARACTER SET utf8mb4
    COLLATE utf8mb4_unicode_ci;
USE real_estate_viewing_variant17;

-- 1. Таблица клиентов
CREATE TABLE clients (
    client_id INT AUTO_INCREMENT PRIMARY KEY,
    last_name VARCHAR(50) NOT NULL,
    first_name VARCHAR(50) NOT NULL,
    phone VARCHAR(20) NOT NULL UNIQUE,
    email VARCHAR(100) NOT NULL UNIQUE,
    CONSTRAINT chk_client_phone CHECK (phone REGEXP '^\\+7[0-9]{10}$')
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

-- 2. Таблица риелторов
CREATE TABLE realtors (
    realtor_id INT AUTO_INCREMENT PRIMARY KEY,
    last_name VARCHAR(50) NOT NULL,
    first_name VARCHAR(50) NOT NULL,
    phone VARCHAR(20) NOT NULL UNIQUE,
    hire_date DATE NOT NULL DEFAULT (CURRENT_DATE)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

-- 3. Таблица объектов недвижимости
CREATE TABLE properties (
    property_id INT AUTO_INCREMENT PRIMARY KEY,
    address VARCHAR(255) NOT NULL,
    property_type ENUM('квартира', 'дом', 'апартаменты') NOT NULL,
    rooms TINYINT UNSIGNED NOT NULL CHECK (rooms > 0),
    area DECIMAL(7,2) NOT NULL CHECK (area > 0),
    price DECIMAL(12,2) NOT NULL CHECK (price > 0),
    realtor_id INT NOT NULL,
    FOREIGN KEY (realtor_id) REFERENCES realtors(realtor_id)
        ON DELETE RESTRICT ON UPDATE CASCADE,
    INDEX idx_property_realtor (realtor_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

-- 4. Таблица записей на просмотр
CREATE TABLE viewings (
    viewing_id INT AUTO_INCREMENT PRIMARY KEY,
    client_id INT NOT NULL,
    property_id INT NOT NULL,
    realtor_id INT NOT NULL,
    viewing_datetime DATETIME NOT NULL,
    status ENUM('запланирован', 'проведён', 'сделка', 'отменён') NOT NULL DEFAULT 'запланирован',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (client_id) REFERENCES clients(client_id)
        ON DELETE RESTRICT ON UPDATE CASCADE,
    FOREIGN KEY (property_id) REFERENCES properties(property_id)
        ON DELETE RESTRICT ON UPDATE CASCADE,
    FOREIGN KEY (realtor_id) REFERENCES realtors(realtor_id)
        ON DELETE RESTRICT ON UPDATE CASCADE,
    UNIQUE KEY unique_viewing_slot (property_id, viewing_datetime),
    INDEX idx_viewing_datetime (viewing_datetime),
    INDEX idx_viewing_client (client_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

-- 5. Таблица избранных объявлений
CREATE TABLE favorites (
    client_id INT NOT NULL,
    property_id INT NOT NULL,
    added_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (client_id, property_id),
    FOREIGN KEY (client_id) REFERENCES clients(client_id)
        ON DELETE CASCADE ON UPDATE CASCADE,
    FOREIGN KEY (property_id) REFERENCES properties(property_id)
        ON DELETE CASCADE ON UPDATE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
